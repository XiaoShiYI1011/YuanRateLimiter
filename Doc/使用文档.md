# YuanRateLimiter  v2.4.4 使用文档

版本：2.4.4  

## 2. 安装

NuGet 安装：

```powershell
NuGet\Install-Package YuanRateLimiter -Version 2.4.4
```

或使用 .NET CLI：

```bash
dotnet add package YuanRateLimiter --version 2.4.4
```

## 3. 快速接入

### 3.1 .NET 6 / Minimal Hosting

`Program.cs`：

```csharp
using YuanRateLimiter;
using YuanRateLimiter.Config;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddControllers();
builder.Services.AddEndpointsApiExplorer();
builder.Services.AddSwaggerGen();

// 使用 Redis：推荐把连接串放在 appsettings.json 或环境变量中，不要写死密码。
builder.Services.AddRateLimiterSetUp(
    config => builder.Configuration.GetSection("RateLimiter").Get<RateLimiterConfig>(),
    builder.Configuration["RedisConfig:Default:ConnectionString"]);

// 不使用 Redis 时，省略第二个参数，使用本机 MemoryCache。
// builder.Services.AddRateLimiterSetUp(
//     config => builder.Configuration.GetSection("RateLimiter").Get<RateLimiterConfig>());

var app = builder.Build();

if (app.Environment.IsDevelopment())
{
    app.UseSwagger();
    app.UseSwaggerUI();
}

app.UseRouting();

app.UseCors(options =>
{
    options.AllowAnyOrigin();
    options.AllowAnyMethod();
    options.AllowAnyHeader();
});

// 建议放在 UseRouting/UseCors 之后，UseAuthorization/MapControllers 之前。
app.UseRateLimitMiddleware();

app.UseAuthorization();
app.MapControllers();
app.Run();
```

### 3.2 .NET 5 / Startup

`Startup.cs`：

```csharp
using Microsoft.AspNetCore.Builder;
using Microsoft.AspNetCore.Hosting;
using Microsoft.Extensions.Configuration;
using Microsoft.Extensions.DependencyInjection;
using Microsoft.Extensions.Hosting;
using YuanRateLimiter;
using YuanRateLimiter.Config;

public class Startup
{
    public IConfiguration Configuration { get; }

    public Startup(IConfiguration configuration)
    {
        Configuration = configuration;
    }

    public void ConfigureServices(IServiceCollection services)
    {
        services.AddControllers();

        services.AddRateLimiterSetUp(
            config => Configuration.GetSection("RateLimiter").Get<RateLimiterConfig>(),
            Configuration["RedisConfig:Default:ConnectionString"]);

        // 使用 MemoryCache：
        // services.AddRateLimiterSetUp(
        //     config => Configuration.GetSection("RateLimiter").Get<RateLimiterConfig>());
    }

    public void Configure(IApplicationBuilder app, IWebHostEnvironment env)
    {
        if (env.IsDevelopment())
        {
            app.UseDeveloperExceptionPage();
        }

        app.UseRouting();
        app.UseRateLimitMiddleware();
        app.UseAuthorization();

        app.UseEndpoints(endpoints =>
        {
            endpoints.MapControllers();
        });
    }
}
```

### 3.3 最小可用配置

这是最小配置，不配置 Redis 时使用 MemoryCache，所有接口共享一个令牌桶。

```json
{
  "RateLimiter": {
    "EnableRateLimiter": true,
    "RateLimiterModel": "TokenBucket",
    "RateLimiterRule": {
      "RateLimiterLogLevel": "All",
      "AllFlowLimiterRule": {
        "Capacity": 100,
        "RateLimit": 20
      }
    }
  }
}
```

含义：

1. 桶容量为 100，允许短时间突发最多消耗 100 个令牌。
2. 每秒补充 20 个令牌，稳定速率约为 20 QPS。
3. 未命中白名单、黑名单后，所有接口共享同一个限流额度。

## 5. 配置项说明

### 5.1 顶层配置

| 配置项 | 类型 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `EnableRateLimiter` | bool | `false` | 是否开启限流。为 `false` 时中间件直接放行 |
| `HttpStatusCode` | int | `429` | 触发限流时返回的状态码 |
| `LimitingMessage` | string | 英文默认提示 | 触发限流时返回的消息 |
| `EnableFallbackCache` | bool | `true` | Redis 不可用时是否降级到 MemoryCache |
| `EnableDoubleWrite` | bool | `false` | Redis 可用时是否同时写 MemoryCache |
| `RedisRetryCount` | int | `3` | 启动时 Redis 连接失败的重试次数，最大会被限制为 10 |
| `RedisRetryDelayMs` | int | `1000` | Redis 重试间隔，最大会被限制为 30000 |
| `CacheKey` | string | `RateLimiterKey` | 限流缓存 key 前缀 |
| `RateLimiterModel` | enum/string | `TokenBucket` | 限流算法：`TokenBucket`、`LeakBucket`、`SlidingWindow` |
| `EnableIpLimiter` | bool | `false` | 是否按客户端 IP 独立限流 |
| `IpWhiteList` | string[] | 空 | 白名单 IP，命中后直接放行 |
| `IpBlackList` | string[] | 空 | 黑名单 IP，命中后直接返回 403 |
| `RateLimiterRule` | object | 默认 All 规则 | 限流规则配置 |

### 5.2 规则配置

| 配置项 | 类型 | 说明 |
| --- | --- | --- |
| `RateLimiterLogLevel` | enum/string | 限流粒度：`All`、`Method`、`Action` |
| `AllFlowLimiterRule` | object | 全接口共享规则 |
| `MethodFlowLimiterRules` | array | 按 HTTP Method 配置规则 |
| `ActionFlowLimiterRules` | array | 按请求路径配置规则 |

### 5.3 基础规则字段

所有规则都继承同一组字段，但不同算法实际使用的字段不同：

| 字段 | TokenBucket | LeakBucket | SlidingWindow | 说明 |
| --- | --- | --- | --- | --- |
| `Capacity` | 使用 | 使用 | 不使用 | 桶容量 |
| `RateLimit` | 使用 | 使用 | 不使用 | 令牌补充速率 / 漏水速率，单位约为每秒 |
| `WindowSize` | 不使用 | 不使用 | 使用 | 滑动窗口大小，单位秒 |
| `MaxRequests` | 不使用 | 不使用 | 使用 | 窗口内最大请求数 |

配置校验器会对空规则、非法枚举、重复 IP、非正数规则值做容错处理，并输出中文 warning。容错是为了防止宿主启动失败，不代表错误配置应该长期保留。

## 6. 限流算法说明

### 6.1 TokenBucket 令牌桶

适合场景：允许短时间突发，但要限制长期平均速率。

配置字段：

```json
{
  "Capacity": 100,
  "RateLimit": 20
}
```

含义：

1. 初始桶容量最多 100 个令牌。
2. 每秒补充约 20 个令牌。
3. 每个请求消耗 1 个令牌。
4. 没有令牌时返回 429。

理论放行量估算：

```text
持续 T 秒内的最大放行量约为 Capacity + RateLimit * T
```

示例：`Capacity = 30`，`RateLimit = 3`，压测 30 秒，理论最大放行量约为：

```text
30 + 3 * 30 = 120
```

### 6.2 LeakBucket 漏桶

适合场景：削平突发流量，让请求以相对稳定的速度通过。

配置字段：

```json
{
  "Capacity": 100,
  "RateLimit": 20
}
```

含义：

1. `Capacity` 表示桶容量。
2. `RateLimit` 表示漏水速率，单位约为每秒。
3. 请求进入桶后水位增加。
4. 水位超过桶容量时返回 429。

### 6.3 SlidingWindow 滑动窗口

适合场景：严格控制任意一段时间窗口内的请求总数。

配置字段：

```json
{
  "WindowSize": 10,
  "MaxRequests": 100
}
```

含义：

1. 任意 10 秒窗口内最多允许 100 个请求。
2. 超过 `MaxRequests` 后返回 429。
3. 当前实现基于缓存列表记录请求时间。

## 7. 限流粒度

### 7.1 All 全接口限流

所有接口共享同一份限流规则。

```json
{
  "RateLimiter": {
    "EnableRateLimiter": true,
    "RateLimiterModel": "TokenBucket",
    "RateLimiterRule": {
      "RateLimiterLogLevel": "All",
      "AllFlowLimiterRule": {
        "Capacity": 100,
        "RateLimit": 20
      }
    }
  }
}
```

适合：

1. 整个服务统一削峰。
2. 简单接口保护。
3. 不需要区分接口和请求方法的系统。

### 7.2 Method 请求方法限流

按 HTTP 方法分别限流，例如 GET 和 POST 使用不同额度。

```json
{
  "RateLimiter": {
    "EnableRateLimiter": true,
    "RateLimiterModel": "TokenBucket",
    "RateLimiterRule": {
      "RateLimiterLogLevel": "Method",
      "AllFlowLimiterRule": {
        "Capacity": 100,
        "RateLimit": 20
      },
      "MethodFlowLimiterRules": [
        {
          "Method": "GET",
          "Capacity": 100,
          "RateLimit": 20
        },
        {
          "Method": "POST",
          "Capacity": 20,
          "RateLimit": 5
        }
      ]
    }
  }
}
```

注意：

1. `Method` 大小写不敏感。
2. 没有配置对应 Method 时，该请求不会命中 Method 规则，会直接放行。
3. 如果 `RateLimiterLogLevel = "Method"` 但没有有效 `MethodFlowLimiterRules`，会回退为 All。

### 7.3 Action 路径限流

按接口路径分别限流。

```json
{
  "RateLimiter": {
    "EnableRateLimiter": true,
    "RateLimiterModel": "TokenBucket",
    "RateLimiterRule": {
      "RateLimiterLogLevel": "Action",
      "AllFlowLimiterRule": {
        "Capacity": 100,
        "RateLimit": 20
      },
      "ActionFlowLimiterRules": [
        {
          "Path": "/api/Test/Test01",
          "Capacity": 10,
          "RateLimit": 1
        },
        {
          "Path": "/api/Test/**",
          "Capacity": 100,
          "RateLimit": 20
        }
      ]
    }
  }
}
```

匹配规则：

1. 先匹配精确路径，再匹配通配路径。
2. 多条通配路径同时命中时，使用配置顺序中的第一条。
3. 路径匹配大小写不敏感。
4. 未命中任何 Action 规则时，请求直接放行。

## 8. Action 路径通配符

| 写法 | 含义 | 示例 |
| --- | --- | --- |
| 精确路径 | 只匹配完全相同路径 | `/api/Test/Test01` |
| `*` | 匹配单个路径段 | `/api/Order/*` 匹配 `/api/Order/1001`，不匹配 `/api/Order/1001/detail` |
| `**` | 匹配多级路径 | `/api/Test/**` 匹配 `/api/Test`、`/api/Test/A`、`/api/Test/A/B` |
| `?` | 匹配单个字符 | `/api/User/?` 匹配 `/api/User/1`，不匹配 `/api/User/12` |
| `{param}` | 匹配一个路径段 | `/api/Product/{id}` 匹配 `/api/Product/1001` |

推荐写法：

```json
"ActionFlowLimiterRules": [
  {
    "Path": "/api/Order/Create",
    "Capacity": 10,
    "RateLimit": 2
  },
  {
    "Path": "/api/Order/**",
    "Capacity": 100,
    "RateLimit": 20
  }
]
```

这样精确规则会优先命中 `/api/Order/Create`，其他订单接口走 `/api/Order/**`。

## 9. IP 限流、白名单、黑名单

### 9.1 开启按 IP 限流

```json
{
  "RateLimiter": {
    "EnableIpLimiter": true
  }
}
```

开启后，缓存 key 会带上客户端 IP，不同 IP 独立计算额度。

适合：

1. 防止单个客户端刷接口。
2. 公共接口按用户来源削峰。
3. 网关后按真实客户端 IP 控制请求量。

不适合：

1. 所有用户都经过同一个 NAT 出口，且你希望区分真实用户。
2. 代理头不可信，客户端可以伪造 `X-Forwarded-For`。

### 9.2 白名单

白名单 IP 不受限流影响，直接进入后续中间件。

```json
{
  "RateLimiter": {
    "IpWhiteList": [ "127.0.0.1", "10.0.0.10" ]
  }
}
```

### 9.3 黑名单

黑名单 IP 直接返回 403。

```json
{
  "RateLimiter": {
    "IpBlackList": [ "203.0.113.200" ]
  }
}
```

返回体：

```json
{
  "statusCode": 403,
  "message": "当前Ip被禁止访问"
}
```

注意：

1. 黑白名单使用精确匹配，不是包含匹配。
2. 同一个 IP 不应同时配置在白名单和黑名单中。当前执行顺序是先检查白名单，再检查黑名单。

## 10. 客户端 IP 获取逻辑

中间件通过 `IPUtil.GetClientIPv4` 获取客户端 IP，优先级为：

1. `X-Real-IP`
2. `X-Forwarded-For` 中第一个非 loopback IP
3. `RemoteIpAddress.MapToIPv4()`

如果项目部署在 Nginx、SLB、Ingress、API Gateway 后面，必须正确配置代理头。

Nginx 示例：

```nginx
location / {
    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_pass http://your_upstream;
}
```

安全建议：

1. 不要直接信任公网客户端传入的 `X-Real-IP` 或 `X-Forwarded-For`。
2. 应由可信反向代理覆盖或追加代理头。
3. 只允许应用接收来自可信负载均衡的流量。

## 11. Redis 与缓存模式

### 11.1 MemoryCache 模式

调用 `AddRateLimiterSetUp` 时不传 Redis 连接串：

```csharp
services.AddRateLimiterSetUp(
    config => Configuration.GetSection("RateLimiter").Get<RateLimiterConfig>());
```

特点：

1. 简单，无外部依赖。
2. 只在当前进程内生效。
3. 多实例之间不共享状态。
4. 适合本地开发、单实例服务、临时保护。

### 11.2 Redis 模式

调用 `AddRateLimiterSetUp` 时传入 Redis 连接串：

```csharp
services.AddRateLimiterSetUp(
    config => Configuration.GetSection("RateLimiter").Get<RateLimiterConfig>(),
    Configuration["RedisConfig:Default:ConnectionString"]);
```

特点：

1. 多实例可共享限流状态。
2. Redis 不可用时，根据 `EnableFallbackCache` 决定是否降级到内存。

### 11.3 Hybrid 降级

默认：

```json
{
  "EnableFallbackCache": true,
  "EnableDoubleWrite": false
}
```

含义：

1. Redis 可用时优先使用 Redis。
2. Redis 操作失败时，如果 `EnableFallbackCache = true`，会降级到 MemoryCache。
3. 如果 `EnableDoubleWrite = true`，Redis 写入时也会写 MemoryCache。

生产建议：

| 场景 | 建议 |
| --- | --- |
| 普通削峰 | `EnableFallbackCache = true` |
| 强约束限流 | 不要简单降级放宽，应设计 fail-closed 或降级策略 |
| 高并发生产 | 谨慎开启 `EnableDoubleWrite`，避免内存占用和写放大 |

## 12. 响应格式

触发限流时，默认返回 429：

```json
{
  "statusCode": 429,
  "message": "请求过于频繁，请稍后再试。"
}
```

命中黑名单时返回 403：

```json
{
  "statusCode": 403,
  "message": "当前Ip被禁止访问"
}
```

响应类型：

```text
application/json;charset=utf-8
```

## 16. 常见问题

### 16.1 为什么配置了限流但没有生效？

请检查：

1. `EnableRateLimiter` 是否为 `true`。
2. 是否调用了 `app.UseRateLimitMiddleware()`。
3. 中间件顺序是否正确。
4. `RateLimiterLogLevel` 是否与规则数组匹配。
5. Action 路径是否和实际请求路径一致。
6. 请求 IP 是否在白名单里。

### 16.2 为什么 Action 规则不命中？

请检查：

1. `RateLimiterLogLevel` 是否为 `Action`。
2. `ActionFlowLimiterRules` 是否非空。
3. `Path` 是否以 `/` 开头。
4. 是否被更前面的精确路径或通配路径匹配。
5. 实际请求路径是否包含路由前缀。

### 16.3 为什么黑名单不生效？

请检查：

1. 中间件获取到的客户端 IP 是什么。
2. 反向代理是否覆盖了 `X-Real-IP`。
3. `IpBlackList` 是否精确等于客户端 IP。
4. 同一 IP 是否也配置到了白名单。

### 16.4 Redis 不可用时会怎样？

如果配置了 Redis 且 `EnableFallbackCache = true`，Redis 操作失败时会尝试降级到 MemoryCache。此时多实例之间不再共享限流状态。

如果业务要求严格全局限流，不应简单依赖 MemoryCache 降级，应提前设计 Redis 故障时的处理策略。

### 16.5 `http_req_failed` 很高是否一定代表服务失败？

不一定。k6 会把 429 也计入失败率。限流测试中 429 是预期结果，应重点看：

1. 200 数量是否符合理论放行量。
2. 429 是否稳定返回。
3. 5xx 是否为 0。
4. dropped iterations 是否为 0。
